<objective>
Close the backend half of the pre-battle-hang blocker surfaced in UAT Test 7 (see `.planning/debug/pre-battle-hang-after-wager.md`, status: root_cause_found). Plans 04-10 + 04-11 closed the frontend UX portion (StuckPane + 45s watchdog + wager-advance cache fix). The backend-side recovery — BattleRoom DO pool-timeout alarm + workflow-started observability + host retry endpoint — was explicitly deferred in Plan 04-10 and is now the lone missing plank.

This plan is BACKEND ONLY:

1. **Schema — observability column.** Add nullable `workflow_started_at INTEGER` to `battle_pool_topics`. Stamped by the workflow's first step. Nulled by the retry endpoint before re-scheduling. Lets the retry endpoint distinguish "silently-dropped (never started)" from "in-flight (slow but running)."

2. **Workflow — observability stamp.** New exported helpers `markWorkflowStarted` + `nullWorkflowStartedAt`. New `record-workflow-started` step placed BEFORE the existing `generate-battle-questions` step, wrapping `markWorkflowStarted`.

3. **BattleRoom DO — pool-timeout alarm.** Schedule a 60s alarm at the END of `opAttachGuest` (after the existing lobby-alarm delete). The existing `alarm()` switch's no-op `case "pre-battle":` branch becomes the pool-timeout branch: reads `battles.poolTopicId` → `battle_pool_topics.status`; if `'generating'`, calls `markPoolTopicFailed`. `opStartBattle` already calls `deleteAlarm()` so the alarm cannot fire after transition to active. Single-alarm-per-DO invariant preserved.

4. **HTTP — host-only retry endpoint.** `POST /api/battle/:id/pool/retry` with the full 403/404/409/200/202 contract. Guarded by authGuard (inherited) + sanitize + `battleJoinRateLimit` (10/min). Host-only via the existing `battle.hostId !== userId → 403` pattern. ZERO request body — topic is re-read from `battle_pool_topics.topic` so there is no user-controllable topic string (no prompt-injection vector).

This plan does NOT touch the frontend. Plan 04-13 wires the StuckPane retry CTA against this endpoint. Plan 04-14 lands the regression tests.

Purpose: Guarantee a battle cannot be stuck on `poolStatus='generating'` longer than 60s regardless of Cloudflare Workflows runtime behavior. Give the host a deterministic recovery path without starting a fresh battle.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client → Worker HTTP (POST /api/battle/:id/pool/retry) | Authenticated session cookie crosses here; Hono route handler is the trust boundary. All validation, host-check, and rate-limit enforcement happens above the DB write. |
| Worker → D1 (battle_pool_topics reads/writes) | Drizzle-typed queries; integrity is enforced by Drizzle schema + existing partial UNIQUE index on join_code. New nullable column has no integrity implications. |
| Worker → Cloudflare Workflows (`env.BATTLE_QUESTION_WORKFLOW.create`) | Workflow scheduler accepts a `params.topic` string that Llama 3.3 ultimately sees. This plan's retry endpoint re-reads topic from `battle_pool_topics.topic` (a column whose content was validated by `assertTopicSafe` when the battle was created) — never from the retry request body. |
| BattleRoom DO → D1 (alarm-branch reads + markPoolTopicFailed) | DO runs in a single-threaded context per battleId; alarm-branch DB lookups are bounded to the owning battle via `cfg.battleId` from ctx.storage. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-gap-10 | I (Info disclosure / IDOR) | POST /api/battle/:id/pool/retry | mitigate | Host-only enforcement via `battle.hostId !== userId → return c.text("Forbidden", 403)` — generic text body indistinguishable from "no such battle" (mirrors existing /:id/cancel pattern at line 714 of battle.ts). authGuard is inherited from `battleRoutes.use("/*", …)`. Non-host guests attempting retry receive 403 with no payload that leaks battle existence or host identity. |
| T-04-gap-11 | D (DoS — thundering herd on flaky pool) | POST /api/battle/:id/pool/retry | mitigate | Two-layer mitigation: (1) `battleJoinRateLimit` (10/min per IP, already bound) caps total retry volume; (2) the `workflow_started_at` in-flight check returns 409 when the current workflow run is < 60s old. A host spamming the button cannot fire multiple parallel workflows for the same poolTopicId. The 60s window is deliberately aligned with `POOL_TIMEOUT_MS` so a retry becomes eligible EXACTLY when the DO alarm would force-fail the pool. |
| T-04-gap-12 | T (Tampering via prompt injection) + I (info disclosure via malicious topic) | POST /api/battle/:id/pool/retry | mitigate | Endpoint accepts ZERO request body. `sanitize` middleware is applied defensively. The retry path re-reads `battle_pool_topics.topic` (a column written ONLY via `findOrQueueTopic` after passing `assertTopicSafe`). There is NO code path by which a malicious retry caller can influence the topic string the workflow ultimately feeds to Llama 3.3 or Vectorize. This eliminates prompt-injection and pool-poisoning vectors at the endpoint level. |
| T-04-gap-11-alarm | D (DoS — DO alarm loop / storage hammering) | BattleRoom DO pool-timeout alarm | mitigate | One-shot alarm: `alarm()` does NOT re-schedule after firing. `opStartBattle`'s existing `deleteAlarm()` covers the transition out of pre-battle. Per-DO serialization (single-threaded input gate) guarantees no concurrent alarm + handler race. `markPoolTopicFailed` uses `INSERT OR IGNORE`-safe UPDATE semantics (no partial-write risk). Error path in the alarm branch is a best-effort `console.error` — never throws. |
| T-04-gap-14 | S (Spoofing via workflow id collision) | env.BATTLE_QUESTION_WORKFLOW.create re-fire | accept | Rationale: Cloudflare Workflows keys runs by `id` — re-creating with the same `poolTopicId` either re-runs the body (documented behavior) OR rejects if the runtime enforces single-instance-per-id. If reject, the existing `try/catch` in the workflow's outer path (gap 04-10) + the DO pool-timeout alarm together still converge to `poolStatus='failed'` within 60s. No attacker control over `poolTopicId` (it's derived from the battles row, not request input). Plan 04-14's integration test will surface this if it becomes an observable hard-fail; at that point we'll wrap `.create` in a scoped try/catch that swallows "instance already exists" errors specifically. |
